# MQ 消费 poison pill 死信与重试兜底 Brief

## 1. 需求摘要

- **做什么**：给 MQ 消费失败路径加上"最大重试次数 + 死信兜底 + 精确位点提交"，消除单条坏消息无限重投、堵死整个 partition 的问题（GitHub issue #22，P0）。
- **为什么做**：当前 Kafka 消费回调抛异常时只打日志、不提交 offset，没有重试上限也没有死信队列。典型触发是"解析其实成功了，但回发 parse_result 通知的 MQ 临时不可用"——程序判定失败并无限重投，同 partition 后续所有解析任务被永久阻塞，且故障无法自愈。
- **本次不做**：不改解析/分块/向量化等业务逻辑；不改 parse_result 消息契约本身；不引入新的 MQ 厂商；不做消费端全局限流/削峰。

## 2. 业务流程

### 2.1 主流程图

```mermaid
flowchart TD
    A["从 MQ 取出解析任务消息"] --> B["执行解析 Pipeline"]
    B --> C{"回调结果"}
    C -->|成功| D["精确提交该 partition 位点"]
    C -->|终态失败<br/>(重试也没用)| E["直接投递死信队列"]
    C -->|可重试失败<br/>(如结果通知 MQ 暂不可用)| F{"重试次数<上限?"}
    F -->|是| G["重投本条消息<br/>(不提交位点)"]
    G --> B
    F -->|否| E
    E --> H["提交位点, partition 继续前进"]
    D --> I["处理下一条"]
    H --> I
```

流程图只画主链路与关键分支。死信队列里的消息后续由人工/运维单独排查，不在本次自动回灌。

### 2.2 流程详解

**取消息与执行**：消费者从解析任务 topic 取出一条消息，交给 `ParseTaskPipeline` 执行（解析→分块→向量化→ES 入库→回发 parse_result 通知）。这一段沿用现有逻辑，不改。

**成功路径**：Pipeline 正常返回，消费者**只提交当前这一条消息所在 (topic, partition) 的精确位点**。这是本次的关键修正点——当前实现调用的是无参提交，会把消费者组在所有已分配 partition 上的"已读位置"一起提交；一旦坏消息卡在中间、后面消息先成功并触发提交，坏消息会被**静默跳过当作没发生（丢数据且无人知晓）**。所以"精确提交"不是优化项，而是其它兜底逻辑能正确生效的前提。

**终态失败路径（重试无意义）**：Pipeline 抛出的异常属于"重试一万次也一样"的类型（如文档本身损坏、解析逻辑 bug、未知内部错误），消费者不再重投，直接把原始消息连同失败原因、来源元数据投递到**死信队列**，然后提交位点让 partition 继续前进。注意：Pipeline 内部对"普通解析失败"已经做了标记失败终态 + 回发失败通知并正常返回（不抛异常），这类不会走到这里；走到这里的是**逃出 Pipeline 兜底之外的异常**（例如兜底逻辑里写库时 DB 抖动又抛了非预期异常）。

**可重试失败路径**：Pipeline 抛出的异常属于"暂时性、过会儿可能就好"的类型，目前唯一明确的来源是 `ParseResultNotificationError`——解析终态已确定，只是回发 parse_result 通知的那条 MQ 发不出去，Pipeline 故意抛出让消息被重投补发。消费者对这类异常做**有限次重投**，每次重试之间**固定间隔退避**（给短暂性故障留恢复窗口）：未达上限就退避后重新执行（不提交位点），达到上限后**降级为死信兜底**并提交位点。单条消息最长阻塞该 partition 的时间 ≈ 退避间隔 × 最大重试次数，二者均可配置。

**死信后的去向**：进入死信队列的消息记录足够的排查信息（原 topic、失败异常、累计重试次数、关键业务标识）。本次只做"进死信 + partition 放行"，不做自动重放；重放/补偿由运维按需手工处理（与现有 chunk 索引重试机制定位不同，见第 4 章风险）。

**RabbitMQ 对齐**：项目同时支持 RabbitMQ，其消费实现用 `message.process()` 上下文管理器，回调异常会自动 nack 重入队，存在**完全相同**的无限重投缺陷。**本次纳入范围**：用 RabbitMQ 死信交换器（DLX）+ 消息计数实现与 Kafka 等价的有限重试 + 死信兜底，保证厂商切换后行为一致、无遗留 P0 洞。

## 3. 核心模块与实现思路

### 3.1 异常分类（Pipeline 层）

- **位置**：`src/core/pipeline/parse_task/`（`notifier.py` 已有的异常定义处，及 Pipeline 抛错路径）。
- **职责**：让消费端能区分"值得重试的暂时性失败"与"重试无意义的终态失败"，这是 adapter 做有限重试决策的依据。
- **实现思路**：
  - 引入一个"可重试异常"语义基类，把现有的 `ParseResultNotificationError` 归入该语义（它的含义正是"终态已确定、只是结果通知没送出去，值得补发"）。
  - Pipeline 现有的兜底 `except` 块逻辑保持不变（普通解析失败仍标记终态、回发失败通知、正常返回，不抛出）；变更只在于：从 Pipeline 真正"逃出去"的异常被消费端按"是否可重试"分流。
- **关键决策**：用异常类型而非返回码传递可重试性——因为 poison pill 恰恰发生在"异常逃出正常返回路径"的场景，返回码覆盖不到；异常基类能同时覆盖现有 `ParseResultNotificationError` 和未来新增的暂时性失败。

### 3.2 有限重试 + 死信兜底 + 精确位点提交（Kafka Adapter 层）

- **位置**：`src/core/mq/vendors/kafka/kafka_adapter.py` 的 `KafkaReceiver._consume_loop`（即 issue 指出的 339-349 行）。
- **职责**：把"失败即不提交、无限重投"改造为"有限重试 → 超限进死信 → 放行 partition"，并把位点提交从"全组无参提交"改为"按当前消息的 (topic, partition) 精确提交"。
- **实现思路**：
  - 维护一个按 (topic, partition, offset) 计数的重试计数器，回调失败时判断异常是否可重试、是否超过上限。
  - 未超限且可重试：不提交位点，让该条消息被重新处理（需修正当前实现——现在不提交但迭代器仍前移，并非真正"重投"，要让失败消息确实被再次消费）。
  - 超限、或异常不可重试：把原始消息体 + 元数据（原 topic、异常摘要、累计重试次数、消息 key）投递到死信 topic，成功投递后提交位点，清理该消息的计数。
  - 死信投递复用现有发送能力（`MQFactory.get_sender()` / `MQService`），由工厂在装配 receiver 时注入"死信投递回调"，使编排逻辑不写死在 Kafka 适配器内部、保持 `IMQReceiver` 接口的厂商中立。
  - 重试计数仅存进程内存（已确认）：只用于遏制单进程内的持续打转；进程重启 / rebalance 后计数清零、从上次提交位点重放，可接受。
  - parse_result 通知重试耗尽后进死信即终止（已确认）：不再额外保证终态通知最终送达下游，重放由运维人工处理。
- **关键决策**：
  - 死信兜底放在 **adapter/消费框架层**而非业务回调里——保证不论哪个业务消费者、哪种异常都有统一的最后防线，符合现有"框架层负责收发、业务层只写回调"的分层。
  - 顺带强制修复**精确位点提交**，否则有限重试与死信逻辑会被"无参提交静默跳过坏消息"的旧行为架空。

### 3.3 配置项与 Topic 装配

- **位置**：`src/config.py`（MQ 配置区，参考已有 `CHUNK_INDEX_RETRY_LIMIT` 的写法）、`.env.example`，以及 `src/core/mq/topic_admin.py`（Kafka topic 规格注册处）。
- **职责**：把最大重试次数、重试退避间隔、死信 topic 命名做成可配置；并由应用启动自动装配死信 topic / 死信交换器。
- **实现思路**：
  - 新增最大重试次数、固定退避间隔、死信 topic 命名规则（**原 topic + 固定后缀 `.DLT`**，已确认）等配置项，集中在 `Settings` 加载，不硬编码。**死信兜底恒启用，不提供关闭开关**（已确认）——避免出现"关闭死信后超限如何处理"的二义语义。
  - 死信 topic 纳入现有 topic 装配清单（`build_default_topic_specs` / `ensure_topics`）由应用启动自动创建；RabbitMQ 侧在启动时声明对应死信交换器/队列。均做幂等声明，避免运行时首次投递因目标不存在而失败、回退到原 poison 行为（已确认采用自动创建）。
- **关键决策**：配置项命名与默认值对齐现有 `CHUNK_INDEX_RETRY_*` 风格，保持运维心智一致。

### 3.4 RabbitMQ 对齐（本次范围内）

- **位置**：`src/core/mq/vendors/rabbitmq_adapter.py` 的 `RabbitMQReceiver`。
- **职责**：用 RabbitMQ 原生机制（死信交换器 DLX + 投递次数/header 计数）实现与 Kafka 等价的"有限重试 + 死信兜底"。
- **实现思路**：失败不再无条件自动 nack-requeue；引入按消息计数，超限后 reject 到死信交换器；正常失败在上限内有限重入队；可重试/终态异常的分流口径与 Kafka 一致。
- **关键决策**：与 Kafka 行为对齐（同样的最大重试次数语义、同样的可重试异常判定），避免两套厂商行为漂移。

## 4. 风险与不确定性

| 风险 / 问题 | 触发条件 | 影响 | 当前判断 / 应对方向 |
| :--- | :--- | :--- | :--- |
| 重试计数仅存进程内存（已决策接受） | 进程重启 / 消费者组 rebalance | 计数清零，从上次提交位点重放，"可重试"消息跨重启可能再走一轮上限内重试 | 已确认接受：仅遏制单进程持续打转，跨重启再试一轮可接受，不引入持久化计数 |
| parse_result 通知最终进死信 = 下游 Java 收不到该任务终态（已决策接受） | parse_result MQ 长时间不可用，重试耗尽 | 解析终态已定，业务系统状态需运维介入处理 | 已确认接受：进死信即终止，不引入终态通知补发链路，重放由运维人工处理 |
| 同步重试期间阻塞该 partition（已决策） | 可重试异常持续在上限内重试，带固定间隔退避 | 该 partition 吞吐下降，最长阻塞 ≈ 退避间隔 × 最大重试次数 | 已确认采用固定间隔退避；阻塞上限可控且可配置，超限即进死信放行 |
| 死信 topic / 交换器未创建（已决策） | 首次触发死信投递时目标不存在 | 死信投递失败，兜底再失败可能回到原 poison 行为 | 已确认应用启动自动幂等创建（Kafka 纳入 topic_admin，RabbitMQ 启动声明 DLX） |
| 精确位点提交改造影响现有提交语义 | 所有正常消费路径 | 改不对会引入重复消费或漏提交 | 必做项；以"按当前消息所属 TopicPartition 提交"为准，配套测试覆盖成功/失败/混合顺序 |
