# 文档切片模块技术文档

## 1. 模块概览

### 1.1 文档说明

本文档描述项目中 `markdown_parser` 与 `splitter` 两个模块的技术设计，重点说明：

- 模块目标与职责边界
- 数据模型与处理链路
- 结构分片与语义分片策略
- 关键设计特点与适用边界

本文档不展开接口签名、参数列表和测试执行细节，这些内容分别放在接口文档与测试文档中维护。

对应代码路径：

- `src/core/markdown_parser`
- `src/core/splitter`

### 1.2 模块目标

这两个模块共同承担 RAG 文档预处理链路中的“结构化解析”和“可检索分片”职责。

总体目标有四点：

1. 将原始 Markdown 文本解析为稳定、可消费的结构化元素序列。
2. 在尽量保留文档语义结构的前提下，生成适合检索的 chunk。
3. 为 embedding、向量库入库、检索结果解释保留充足元数据。
4. 保持 `markdown_parser` 与 `splitter` 解耦，让分片模块只依赖解析结果契约，而不依赖解析器内部实现。

### 1.3 总体处理链路

整体处理流程如下：

```text
Raw Markdown
    |
    v
MarkdownParser
    |
    +-- ImageExtractor -> ImageRef[]
    +-- MarkdownScanner -> MarkdownElement[]
    |
    v
ParseResult
    |
    +-- ImageDescriber   (optional)
    +-- TableDescriber   (optional)
    |
    v
ChunkingEngine
    |
    +-- ASTAwareChunker / StructuredSemanticChunker
    |
    v
Chunk[]
    |
    +-- ChunkEmbeddingPipeline (optional)
    |
    v
EmbeddedChunk[]
```

从职责上看：

- `markdown_parser` 负责“把文档解析成结构”
- `splitter` 负责“把结构切成可检索块”
- `embedding_pipeline` 负责“把最终 chunk 变成向量化产物”

### 1.4 职责边界

#### 1.4.1 `markdown_parser`

`markdown_parser` 负责：

- 识别 Markdown 中的块级结构
- 提取图片引用、表格内容、标题层级、代码块语言等结构信息
- 输出统一的 `ParseResult`
- 在可选情况下，将图片描述、表格总结回填进解析结果

`markdown_parser` 不负责：

- chunk 切分策略
- token 长度控制
- 语义切片
- 向量化
- 数据库存储

#### 1.4.2 `splitter`

`splitter` 负责：

- 消费 `ParseResult` 或 `MarkdownElement[]`
- 按结构规则和语义规则生成 `Chunk`
- 维护 chunk 的顺序、上下文和元数据
- 在可选情况下完成最终 chunk 向量化

`splitter` 不负责：

- Markdown 语法识别细节
- 图片下载或 VLM 推理
- 表格总结生成
- 数据库适配与存储

#### 1.4.3 封装原则

本模块采用如下封装原则：

- `splitter` 只消费 `markdown_parser` 的公开输出契约
- `splitter` 不依赖 `markdown_parser` 的内部扫描过程
- `markdown_parser` 不感知 `splitter` 的 chunk 策略

这意味着：

- 可以替换解析器实现，但只要 `ParseResult` 契约不变，`splitter` 无需改动
- 可以替换 chunker 策略，但只要输入仍是 `MarkdownElement[]`，`markdown_parser` 无需改动

## 2. `markdown_parser` 技术设计

### 2.1 模块组成

`markdown_parser` 由以下组件构成：

- `models.py`
  - `ElementType`
  - `MarkdownElement`
  - `ImageRef`
  - `TableRef`
  - `ParseResult`
- `scanner.py`
  - `MarkdownScanner`
- `image_extractor.py`
  - `ImageExtractor`
- `parser.py`
  - `MarkdownParser`
- `llm_integration.py`
  - `VisionClient`
  - `ImageDescriber`
  - `TableClient`
  - `TableDescriber`

### 2.2 核心数据模型

#### 2.2.1 `ElementType`

支持的 Markdown 元素类型包括：

- `HEADING`
- `PARAGRAPH`
- `CODE_BLOCK`
- `LIST`
- `BLOCKQUOTE`
- `TABLE`
- `IMAGE`
- `HORIZONTAL_RULE`
- `FRONT_MATTER`
- `MATH_BLOCK`

这个枚举是 parser 与 splitter 之间最重要的结构契约之一。

#### 2.2.2 `MarkdownElement`

`MarkdownElement` 是扫描器输出的最小结构单元，包含：

- `type`
- `content`
- `start_line`
- `end_line`
- `metadata`

其中 `metadata` 用于存储不同元素的附加信息，例如：

- `heading_level`
- `heading_text`
- `language`
- `url`
- `alt`

#### 2.2.3 `ParseResult`

`ParseResult` 是 `MarkdownParser` 的统一输出对象，包含：

- `elements`
- `tables`
- `images`
- `source_file`
- `remainder`

这里的设计意图很明确：

- `elements` 给 `splitter` 消费
- `tables` 与 `images` 给可选的 LLM 增强流程使用
- `source_file` 用于元数据透传

### 2.3 `MarkdownScanner` 扫描策略

`MarkdownScanner` 是实际执行块级识别的核心组件，采用逐行扫描模型。

识别顺序大致如下：

1. Front Matter
2. 空行跳过
3. Horizontal Rule
4. Heading
5. Code Block
6. Math Block
7. Standalone Image
8. Table
9. List
10. Blockquote
11. Paragraph

这个顺序的设计意义在于：

- 高优先级结构先被捕获，避免被段落兜底逻辑吞掉
- 表格必须在段落前识别，否则容易和普通文本混淆
- 独立图片行必须优先于段落识别，否则无法单独建块

### 2.4 关键实现特点

#### 2.4.1 段落以空行为硬边界

段落提取逻辑在遇到空行时立即截断，不跨空白区域合并段落。

这样做的好处是：

- 更符合标准 Markdown 语义
- 不会因为表格剔除或结构块存在而错误合并远距离文本

#### 2.4.2 表格原生识别

表格不是通过后处理猜测，而是在扫描阶段直接识别：

- 行含 `|`
- 下一行符合表头分隔线格式

这样能保证：

- 表格位置精确
- 表格不会被段落吞并
- 可以直接单独切块或交给 LLM 做总结

#### 2.4.3 图片双通道识别

图片识别分为两层：

- `ImageExtractor` 提取所有图片引用
- `MarkdownScanner` 对独立图片行直接生成 `IMAGE` 类型元素

这样 parser 层既能保留图片引用列表，又能在结构层把图片作为独立元素暴露给 splitter。

### 2.5 `MarkdownParser` 的编排职责

`MarkdownParser` 负责统一编排解析过程：

1. 提取图片引用
2. 扫描 Markdown 结构
3. 从 `elements` 中收集 `TableRef`
4. 产出 `ParseResult`

它只负责组织解析步骤和生成统一结果对象，不负责 chunk 策略和下游向量化。

### 2.6 LLM 增强集成层

`llm_integration.py` 不是 parser 的必经链路，而是可选增强层。

提供两类扩展：

- `ImageDescriber`
- `TableDescriber`

它们的工作方式是：

1. 接收 `ParseResult`
2. 调用外部开发者实现的抽象客户端
3. 把生成的自然语言描述回填到对应元素内容中

这层设计的价值在于：

- 让 parser 保持纯结构解析能力
- 让图片理解、表格总结等高成本能力变成可插拔增强

### 2.7 `markdown_parser` 的非目标

该模块刻意不做以下事情：

- 行内语法级 AST 构建
- 超链接知识抽取
- 图片真实下载与存储
- 文档切片
- 语义分段

因此它更接近“结构化块提取器”，而不是完整 Markdown 编译器。

## 3. `splitter` 技术设计

### 3.1 模块组成

`splitter` 由以下组件构成：

- `models.py`
  - `Chunk`
  - `EmbeddedChunk`
  - `EmbeddingPipelineStats`
- `base.py`
  - `BaseChunker`
- `rule_chunker.py`
  - `ASTAwareChunker`
- `semantic_chunker.py`
  - `PercentileSemanticChunker`
  - `SemanticChunkingStats`
- `pipeline_chunker.py`
  - `StructuredSemanticChunker`
- `chunking_engine.py`
  - `ChunkingEngine`
- `embedding_pipeline.py`
  - `ChunkEmbeddingPipeline`

### 3.2 核心数据模型

#### 3.2.1 `Chunk`

`Chunk` 是分片阶段的标准产物，包含：

- `content`
- `start_line`
- `end_line`
- `metadata`

它是 embedding、向量库接入和检索结果展示的核心中间对象。

#### 3.2.2 `EmbeddedChunk`

`EmbeddedChunk` 在 `Chunk` 基础上增加：

- `embedding`
- `embedding_model`
- `cached`

因此最终结果不是“只有向量”，而是“文本 + 元数据 + 向量”的组合结构。

### 3.3 抽象层与编排层

#### 3.3.1 `BaseChunker`

`BaseChunker` 定义了所有分片策略的统一契约，包括同步入口、异步入口以及直接消费 `ParseResult` 的便捷入口。

这个抽象层的作用是：

- 允许多种 chunk 策略共存
- 让 `ChunkingEngine` 只依赖抽象接口
- 统一同步与异步调用方式

#### 3.3.2 `ChunkingEngine`

`ChunkingEngine` 是 splitter 的主入口，负责把：

- `MarkdownParser`
- `BaseChunker`

编排成完整处理链。

它最重要的价值是：

- 对上层调用者隐藏 parser 与 chunker 的组合细节
- 允许直接从文本、文件或中间 `ParseResult` 进入分片流程

### 3.4 第一阶段：结构规则分片

`ASTAwareChunker` 负责第一阶段结构切分。

它的核心规则如下：

#### 3.4.1 噪音元素过滤

以下元素不会进入最终检索文本：

- `FRONT_MATTER`
- `HORIZONTAL_RULE`

#### 3.4.2 标题强边界

标题用于维护 `heading_trail`。

规则如下：

- `h1 ~ h3` 触发强切分
- `h4 ~ h6` 作为普通正文元素继续并入当前块

这样可以在“章节语义”与“块数量控制”之间取得平衡。

#### 3.4.3 结构化实体隔离

以下元素会被强制单独成块：

- `CODE_BLOCK`
- `MATH_BLOCK`
- `TABLE`
- `IMAGE`

这保证了：

- 代码块不会被普通正文夹断
- 表格不会被语义切片拆碎
- 图片块可单独保留和扩展

#### 3.4.4 普通正文合并

段落、列表、引用、低级标题等普通内容会先进入 buffer，再合并为一个规则块。

### 3.5 第二阶段：语义切分

“语义切分”是 splitter 中最重要的创新点。它不是简单地按 token 长度硬切，而是在结构分片之后，对“仍然过长的正文块”做局部语义细分。

其核心目标是解决一个典型问题：

- 固定相似度阈值在不同文体下泛化能力差
- 只按 token 上限硬切会破坏段落之间的语义连续性
- 纯结构切分无法处理超长章节内部的语义跳变

本模块采用“95 分位数动态阈值”的自适应切分策略。

#### 3.5.1 设计动机

在真实 Markdown 文档中，文本风格差异很大：

- 教程文档通常语义变化平缓
- 规范文档在章节切换处可能出现明显跳变
- FAQ 或列表文档往往段落很短且切换频繁

如果对所有文档统一使用固定阈值，例如 `0.45`，常见问题有：

- 对高度同质化文档切不动
- 对变化频繁文档切得过碎
- 对不同 embedding 模型适配性差

动态阈值的核心思想是：

- 不预设“什么距离一定算语义断裂”
- 而是先观察文本块内部相邻段落的距离分布
- 再从这个局部统计分布里找出“最异常的跳变点”

这样阈值就不再是全局常数，而是文档局部自适应结果。

#### 3.5.2 算法流程

`PercentileSemanticChunker` 的流程可以拆成四步。

第一步，原子化拆解。

- 优先按段落切分
- 段落过长时降级为按行
- 行仍过长时再降级为按句子

这样做的目的是让 embedding 的基本单元尽量贴近自然语义边界，而不是任意长度片段。

第二步，批量向量化与距离计算。

- 对所有原子单元批量做 embedding
- 计算相邻单元之间的 cosine distance

这里使用“距离”而非“相似度”，是因为距离越大越容易表达语义跳变。

第三步，动态阈值嗅探。

- 对距离序列计算 `percentile`
- 默认使用 95 分位数
- 另外叠加 `min_distance_gate`

最终只有同时满足以下条件的点才会被认为是候选断点：

- 大于该文本块的局部分位阈值
- 大于绝对最小语义距离门槛

这样可以避免两种极端情况：

- 文档高度同质化时被随机噪声触发切分
- 文档本身距离普遍较高时阈值过松导致乱切

第四步，约束执行。

即便遇到候选断点，也不会立刻切开，而是继续叠加工程约束：

- 当前块长度若未达到 `min_chunk_tokens`，忽略这次断点
- 一旦达到 `max_chunk_tokens`，即便没有语义断点也必须强制切分
- 真正切开后，通过 overlap 保留局部连续上下文

这一步非常关键，因为它把“语义合理”变成了“语义合理且工程上可用”。

#### 3.5.3 为什么选 95 分位数

95 分位数不是一个绝对数学真理，而是一个偏工程稳健的选择。

它的意义是：

- 只把最显著的一小部分语义跳变当作候选断点
- 默认偏保守，不轻易把正文切碎
- 对多数教程、说明文、知识库文档更友好

如果分位值过低，例如 70 或 80，通常会出现：

- 候选断点过多
- chunk 数量膨胀
- 上下文被切散

如果分位值过高，例如 99，则又可能出现：

- 断点过少
- 超长语义块切不开

因此 95 分位数本质上是“偏保守但能捕捉显著跳变”的折中。

#### 3.5.4 和固定阈值方案的区别

固定阈值方案的逻辑是：

- 只要相邻段落距离大于某个常数，就切

动态阈值方案的逻辑是：

- 先理解这篇文本内部的距离分布
- 再找出相对异常的跳变点

因此两者最大的差别在于：

- 固定阈值依赖“经验常数”
- 动态阈值依赖“局部统计”

这让该方案对不同文风、不同章节密度、不同 embedding 分布具有更好的自适应性。

#### 3.5.5 工程保护机制

语义切分不是单纯的算法实验实现，而是带有明确工程保护机制。

第一，最小长度保护。

- 即使某个位置语义跳变很大
- 只要当前块还太短，就先不切

这样避免生成大量碎片块。

第二，最大长度阻断。

- 即便文本语义变化平缓
- 一旦超过最大 token 上限，也必须切开

这样避免生成无法入库或不适合检索的超长块。

第三，overlap 保护。

- 真正切块时会保留一部分尾部上下文
- 让后续块不会完全脱离前文

这样可以降低切分对检索连贯性的损失。

第四，fallback。

- embedding 接口失败
- 向量形状异常
- 外部模型短时不可用

这些情况下会自动退化为长度切分，而不是让整个预处理链路失败。

#### 3.5.6 实现边界

语义切分是“局部相邻段落驱动”的实现，不是全局最优分段算法。

它的特点是：

- 成本可控
- 可解释性强
- 容易调参与观察

同时也意味着：

- 它不能回看整个文档做全局最优分段
- 它默认只利用相邻原子单元的局部距离
- 它更适合作为结构分片后的二次细分器，而不是独立全文段落规划器

### 3.6 两阶段组合分片

`StructuredSemanticChunker` 是主实现 chunker。

它把两阶段组合起来：

1. 先按结构切分
2. 只对超长正文 buffer 进入语义细分
3. 对独立结构块保持隔离
4. 对最终 chunk 统一补邻接上下文 overlap

这意味着系统不是“先全文语义切”，而是：

- 先保护 Markdown 结构
- 再对结构块中的超长文本做局部语义优化

这是面向 RAG 场景更稳妥的设计，因为它同时兼顾了：

- 语义边界
- 结构边界
- 检索成本
- 可解释性

### 3.7 Overlap 设计

overlap 不是只作用于语义子块，而是作用于最终 chunk 邻接关系。

由 `StructuredSemanticChunker._apply_neighbor_context()` 统一处理。

策略为：

- 取前一个 chunk 尾部的最多 `N` token
- 取后一个 chunk 头部的最多 `N` token
- 拼接进当前 chunk 内容

这个设计的意义是：

- 图片、表格、公式、代码块等 isolated chunk 也能携带上下文
- 下游检索命中后更容易恢复局部语境

相关元数据包括：

- `context_prev_tokens_applied`
- `context_next_tokens_applied`
- `context_overlap_mode`

### 3.8 向量化阶段设计

`ChunkEmbeddingPipeline` 负责对最终 `Chunk[]` 进行向量化。

其职责包括：

- 调用 `ChunkingEngine` 获取最终 chunk
- 按 `batch_size` 批量请求 embedding
- 对相同文本结果做缓存复用
- 输出 `EmbeddedChunk[]`

这里需要强调一个容易混淆的点：

- 语义切片阶段的 embedding 只用于判断断点
- 最终向量库入库需要的是“最终 chunk 的 embedding”

因此标准流程通常是两次不同目的的向量化：

1. 段落级 embedding，用于切片决策
2. chunk 级 embedding，用于检索索引

缓存键由以下内容生成：

- `embedding_model`
- `chunk.content`

其目标是避免：

- 同模型同文本重复向量化
- 模型切换后错误复用旧向量

### 3.9 元数据设计

`Chunk.metadata` 是开放字典，已经承载了一批检索和入库相关字段。

现有字段包括但不限于：

- `source_file`
- `element_types`
- `chunk_index`
- `heading_trail`
- `split_strategy`
- `semantic_percentile`
- `semantic_threshold`
- `semantic_group_index`
- `semantic_subchunk_index`
- `context_prev_tokens_applied`
- `context_next_tokens_applied`
- `context_overlap_mode`

这意味着 chunk 产物本身已经具备良好的数据库接入前提。

## 4. 模块协作关系

### 4.1 标准链路

标准链路如下：

```text
Markdown 文本
-> MarkdownParser.parse()
-> ParseResult
-> StructuredSemanticChunker / ChunkingEngine
-> Chunk[]
-> ChunkEmbeddingPipeline
-> EmbeddedChunk[]
```

### 4.2 带 LLM 增强的链路

如果启用图片描述和表格总结，则链路变为：

```text
Markdown 文本
-> MarkdownParser.parse()
-> ParseResult
-> ImageDescriber.process()      (optional)
-> TableDescriber.process()      (optional)
-> ChunkingEngine.process_parse_result()
-> Chunk[]
-> ChunkEmbeddingPipeline.process_parse_result()
-> EmbeddedChunk[]
```

这样设计的好处是：

- 增强步骤可插拔
- parser 输出仍是统一契约
- splitter 无需知道增强细节

## 5. 关键特点总结

本模块的关键特点是：

1. 先结构化、后语义化，而不是纯 token 硬切。
2. Markdown 结构块具有明确的保护优先级。
3. 语义切片采用动态阈值而非固定阈值。
4. isolated chunk 也支持最终邻接上下文 overlap。
5. 最终输出同时保留文本、元数据和向量。
6. parser 与 splitter 已按公开契约实现解耦。

## 6. 适用边界

### 6.1 `markdown_parser` 边界

- 主要识别块级结构，不做细粒度行内 AST
- 表格识别以常见 Markdown 表格为主，不追求完整 HTML 表格语义解析
- 图片描述和表格总结依赖外部开发者注入客户端

### 6.2 `splitter` 边界

- 结构切分规则主要围绕 Markdown 文档场景
- 语义切分基于线性相邻段落距离，不是全局最优分段
- overlap 采用文本拼接模式，不是数据库检索时的邻块实时拼接

### 6.3 管线边界

- 向量化阶段负责生成最终检索向量，不承担数据库存储职责
- `EmbeddedChunk` 负责承载“文本 + 元数据 + 向量”，不直接定义具体数据库 schema

## 7. 结论

`markdown_parser` 与 `splitter` 构成了一条完整、解耦清晰的 RAG 预处理链路：

- `markdown_parser` 负责把 Markdown 文档解析成稳定的结构化中间表示
- `splitter` 负责在保护结构语义的基础上产出检索友好的 chunk
- `embedding_pipeline` 负责把最终 chunk 变成向量化产物

这套设计的核心价值不在于“切得多快”，而在于：

- 结构边界可控
- 元数据完整
- 语义切片可配置
- 数据库接入边界清晰

这份文档作为模块归档，记录的是稳定的设计事实、处理边界与核心实现思路。
