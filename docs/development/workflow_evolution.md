# 从口头沟通到 Spec-as-Test：研发协作流程的三次迭代

> 本文记录 toLink-Rag 团队研发协作流程的三次重构，每一次都不是因为追逐"最佳实践"，而是因为上一版流程在真实协作中真的出了事。

---

## 写在前面

我们团队没有专职产品经理，开发者兼任产品角色，大量使用 LLM 辅助开发。这种新型团队结构下，传统的研发流程并不天然合适。

写下这段历史，既是对自己的复盘，也想给和我们境况类似的团队一个参考。

---

## 第一阶段：没有流程，靠会议和口头沟通

### 当时的协作方式

项目早期，团队人数不多，需求来得快：

- 周一会议：业务方简单说一下要做什么
- 散会后：Py 端和 Java 端的开发者各自理解、各自开始动手
- 中间遇到问题：在群里 @ 一下、当面聊几句
- 上线前：互相联调，发现各种不一致再修

没有文档，没有规范。原因不是"我们不想写"，而是"看起来不必要"——大家都在一个会议室、需求迭代快、写文档"耽误效率"。

这套方式撑了大概两个月。

### 出问题了

问题集中在 **Py 后端和 Java 业务系统跨模块协作**的场景。

举一个真实例子。Py 端负责文档解析后通过 MQ 通知 Java 端，Java 端再写入业务库。会议上口头确认的契约是：

> "解析完了 Py 发个消息过来，Java 拿到就入库。"

听起来很清楚。实际开发出来：

- **字段命名漂移**：Py 端发的是 `parsed_at`，Java 端等的是 `complete_time`。两端都觉得"自己的命名更合理"，没人发现已经不一致。
- **状态机理解不一致**：Py 端定义了 `PENDING → PARSING → DONE/FAILED`，Java 端理解的是 `创建 → 完成`，没考虑 `FAILED` 怎么处理，结果失败的任务在 Java 端永远卡在"处理中"。
- **重试语义没对齐**：Py 端会在 MQ 失败时重投，Java 端没做幂等，导致同一条业务数据入库三次。
- **异常字段没约定**：Py 端遇到不可恢复错误时发了 `error_type=permanent`，Java 端从没见过这个字段，直接当成普通错误处理。

上线那周大家加班修了好几天。复盘时发现一个共同点：**没有一个人是"故意"理解错的，每个人都基于自己的会议记忆做了"合理"假设。但合理假设凑在一起，就不合理了**。

### 第一阶段的教训

- 隐式知识不会自己变成共识。
- 口头确认的"清楚"，在两个不同语境的开发者脑子里是两种"清楚"。
- 协作的复杂度不是线性增长——加一个模块、加一个语言端，沟通失败的方式会指数级增长。

## 第二阶段：引入三段式文档流水线

### 我们决定写下来

痛过一次后，团队决定建立正式的研发文档流程。设计了三段式：

```
原始需求
  ↓
pre_requirement_analysis.md   (需求预分析，开发者读)
  ↓
requirement.md (PRD)          (面向 LLM 的结构化需求)
  ↓
technical_design.md           (面向实现的技术方案)
  ↓
Code
```

我们还为每个阶段写了 skill（agent 工作指令），定义了详细的模板：

- **pre_requirement_analysis.md**：9 个一级章节
- **requirement.md**：14 个一级章节
- **technical_design.md**：14 个一级章节

每个文档由对应 skill 生成、由开发者审核、审核通过后再进入下一阶段。看起来非常严谨。

### 它确实解决了一些问题

引入这套流程后，第一阶段那种"字段名漂移"、"状态机理解不一致"的低级失误大幅减少。两端开发者都能从 PRD 里看到统一的状态定义、字段命名、消息契约。**跨模块协作的事故，从一个月好几起降到几乎绝迹**。

这是真实的进步，不能否认。

### 但新问题浮出水面

**痛点 1：三份文档信息重叠严重**

主流程图在三份文档里各画了一次。状态变化在预分析、PRD、TD 里各描述了一次。风险列表也是。业务一变，三处都要改；漏改一处，文档之间就漂移。我们甚至专门加了一条 pre-commit 规则强制 `CLAUDE.md ↔ AGENTS.md` 同步——这正是"用流程压制结构缺陷"的典型信号。

**痛点 2：PRD 的边界尴尬**

PRD 写到一半总会卡住——它既要承担"业务规格"的职责，又被要求"面向 LLM 后续设计"。结果 PRD 里出现了"业务对象的数据可见性"、"中间件边界"、"面向 technical-design 的输入清单"等章节。**它既不是纯业务，也不是技术**，导致后续 TD 沦为"PRD 技术清单的细化"，失去独立设计空间。

**痛点 3：自然语言留下了太多模糊空间**

这是最严重的问题。PRD 里允许写：

> "用户上传 PDF 后，系统应正确处理重复上传场景"

读起来通顺，但留下大量未定义空间：重复上传是返回旧 task_id 还是报错？是否触发 MQ？是否更新时间戳？开发者审核时不一定发现，LLM 拿到这条需求会自行决定。

我们陆续遇到过这样的情况：PRD 写得看似详尽，14 章每章都填得满满当当，开发者也认真审核通过了。但代码写出来上线后，发现**业务行为和我们脑子里以为的不一样**。复盘时翻 PRD，那条规则的描述是"应正确处理"——技术上 LLM 没写错，但它脑补的"正确"和我们脑补的"正确"不是同一件事。

**痛点 4：审核负担过重**

14 章 PRD + 14 章 TD，认真审核一份要 30-60 分钟。开发者互相 review 的意愿急剧下降——"反正下游 skill 会再审一遍"，于是审核变成走过场。

### 第二阶段的教训

- 写下来比不写好，但**写多份重叠的下来**比写一份精炼的差。
- 自然语言的精确度上限就在那里。**纪律压不住歧义**——你可以规定"不许写'适当处理'"，但人会换更隐蔽的模糊措辞。
- 流程的强度不取决于它写得多详细，而取决于它**消除歧义的结构性能力**。

最后一条是我们花了很长时间才想明白的。

## 第三阶段：brief + acceptance.feature + TD

### 一个新的视角

某次复盘时，有人提出一个问题：

> "我们三份文档都在试图描述需求，但描述本身就是模糊的源头。能不能让需求'不是被描述，而是被断言'？"

这句话戳中了所有人。我们重新审视当时的工作流，发现一个之前忽略的事实：**PRD 和 TD 都是给 LLM 看的，且 LLM 实际"消费"它们的方式是"读完之后写代码"**。那么衡量这两份文档是否合格的唯一标准就是：**LLM 写出的代码是不是符合预期**。

而"符合预期"——能不能换种方式表达？能不能让它**机器可验证**？

答案是肯定的，工具叫 Gherkin。

### 新工作流

```
原始需求
  ↓
brief.md              (业务理解 + 模块实现思路，给开发者审阅)
  ↓
acceptance.feature    (Gherkin 验收契约，给开发者审核 + LLM + pytest-bdd)
  ↓
technical_design.md   (改动文件树 + 方法级方案，给 LLM 实现)
  ↓
Code + Tests          (pytest-bdd 直接消费 .feature)
```

三个产物职责零重叠：

| 产物 | 唯一职责 |
|---|---|
| brief.md | 回答 **为什么做** + 涉及哪些模块、各模块实现思路 |
| acceptance.feature | 回答 **什么是做对了**（机器可验证） |
| technical_design.md | 回答 **在哪里做、怎么做** |

### 关键变化 1：把 PRD 替换成可执行的验收契约

原来写：

> "用户上传 PDF 后，系统应正确处理重复上传场景"

现在写：

```gherkin
Scenario: 重复上传同一文件
  Given 已存在 task=T1 hash=abc123 status=PARSED
  When alice 再次提交 PDF hash=abc123
  Then 接口返回 task_id=T1
  And 不创建新任务
  And 不发送 MQ 消息
```

**写到 `Then` 这一刻，你被强制做出决定**——是返回旧 ID 还是报错？是否更新时间戳？在散文 PRD 里你可以含糊带过，在 Gherkin 里你写不出来。**歧义在打字时就卡住，而不是上线后才暴露**。

更进一步：这些 Scenario 直接被 pytest-bdd 编译成测试用例。**"代码符合预期"不再是主观判断，而是测试通过与否**。

### 关键变化 2：brief 从摘要文档变成实现思路文档

新的 brief.md 不追求短，追求"让另一个开发者读完能在脑子里把这次需求跑通"。它包含 5 个固定章节：

1. **需求摘要**：做什么、为什么做、不做什么
2. **业务流程**：Mermaid 主流程图 + 关键节点详解
3. **核心模块与实现思路**：每个涉及的模块独立一段，写位置、职责、复用什么、新增什么、关键决策。**不到代码层**，但要让人能在脑子里跑通。
4. **风险与不确定性**：具体场景，不写泛泛的话
5. **待确认问题**：仅迭代期使用，收敛后删除

brief 是**可迭代的**——开发者审阅有疑问就和 Agent 对话修改，反复直到满意。这正是第二阶段预分析想做但被复杂模板淹没的事。

### 关键变化 3：TD 吸收原 PRD 的技术内容

既然 PRD 不再存在，原 PRD 里的"业务对象数据可见性、系统协作、中间件边界"等内容由 TD 一次成型，不再有"需求级技术边界"和"技术方案"两次描述同一件事的尴尬。

TD 新增的重要约束：**方法级变更总表中每个方法必须关联到至少一条 Scenario**——这是新流程的核心追溯链：**业务规则 → 验收 Scenario → 方法实现 → 测试**。

### 它解决了什么

| 第二阶段痛点 | 在第三阶段如何解决 |
|---|---|
| 三份文档信息重叠 | 三个产物职责切割，零重叠 |
| PRD 边界尴尬 | PRD 这个角色消失，被 .feature 取代 |
| 自然语言模糊 | Gherkin 强制断言，写不出模糊话 |
| 审核负担过重 | 审 20 条 Scenario 比审 14 章 markdown 快 5 倍 |
| LLM 脑补 / 代码偏离预期 | .feature 同时是测试，机器自动验证 |

更重要的是，**它解决了第二阶段没意识到的元问题**：把"代码符合预期"从主观判断变成客观验证。这件事在 LLM 协作密集的团队里，**直接决定了交付质量的下限**。

## 回看三个阶段

| 阶段 | 协作方式 | 主要失败模式 |
|---|---|---|
| 一：口头沟通 | 会议 + 即时通讯 | 跨模块隐式契约失效 |
| 二：三段式文档流水线 | 9 + 14 + 14 章模板 | 信息重叠、自然语言模糊、LLM 脑补 |
| 三：brief + Gherkin + TD | 5 章 brief + 可执行 spec + 技术方案 | （待暴露） |

每个阶段都解决了上一个阶段的问题，同时**引入了上一个阶段看不见的新问题**。这是软件工程协作流程演进的常态——**不存在终极方案，只存在"匹配当前痛点的下一版"**。

### 我们学到的几条原则

**1. 流程的强度不在于详细程度，在于它消除歧义的结构性能力。**

写得越详细越好，是一种直觉但错误的认知。第二阶段 37 章模板没能阻止"代码不符合预期"，第三阶段 5 章 brief + 简洁的 Gherkin 反而做到了。

**2. 能让机器验证的事，就不要交给人。**

人的注意力是稀缺资源。审 14 章 markdown 时漏掉一条规则太正常了；让测试自动验证 20 条断言，几乎不可能漏。

**3. 重构流程的时机，是上一版流程开始用"压制纪律"维持运转的时候。**

当你需要写 pre-commit 强制 A 文档改了 B 也必须改、需要规定"不许使用模糊措辞"、需要 reviewer 反复提醒"请认真审核"——这些都是流程结构在出毛病的信号。**纪律是流程缺陷的临时补丁，不是健康流程的特征**。

**4. 没有专职产品角色的团队，最缺的不是"和别人沟通需求的文档"，是"强迫自己在写代码前把业务想清楚的工具"。**

开发者作为产品角色，最大的失败模式是"业务想到 60% 就开始动手"。文档不能阻止这件事，可执行的 spec 能。

## 给类似团队的建议

如果你的团队也是开发者兼产品角色、大量用 LLM 辅助开发、踩过"代码上线发现业务理解错"的坑：

- **如果还在第一阶段（口头沟通）**：先建立基础文档规范，不必追求精细，**先有比好更重要**。
- **如果在第二阶段（重模板文档）**：开始审视模板间的重叠和自然语言的模糊；不要急着推翻，**先试点一个模块用 Gherkin 写验收**，看团队感受。
- **如果在准备第三阶段**：投入一周建好 pytest-bdd 基础设施，挑下一个新功能完整走一遍，retro 后再决定全面铺开。

不要试图一步到位。我们用了半年多才走完三个阶段，每一步都是被真实痛点推着走的。**没有真实的痛，重构往往会落地为另一种形式主义。**

---

## 结语

回过头看，三个阶段的本质演进可以归结为一句话：

> 从 **"靠人对齐"** 到 **"靠文档对齐"** 到 **"靠契约对齐"**。

文档是人和人之间的协议，契约是人和机器之间的协议。前者依赖人的注意力，后者依赖机器的执行力。**当 LLM 成为协作主体之一，靠人的注意力维持质量已经不够用了**——这是新时代研发协作必须直面的问题。

我们的第三阶段也不是终点。可以预见，未来会有新的痛点暴露——也许是 Gherkin 表达力不够、也许是 brief 的迭代成本仍然偏高、也许是某种我们现在还想不到的失败模式。到那时再迭代就是。

**研发流程不是一次性设计，是一次次和现实碰撞后的演化产物。**
