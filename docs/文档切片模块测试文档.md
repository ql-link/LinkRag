# 文档切片模块测试文档

## 1. 文档说明

本文档描述 `markdown_parser` 与 `splitter` 相关测试的目标、测试分层、测试资源、Mock 策略、覆盖范围与执行方式。

本文档不重复技术文档中的架构设计，也不重复接口文档中的方法签名说明，重点回答两个问题：

- 当前测了什么
- 这些测试如何保证模块行为稳定

## 2. 测试目标

测试目标主要分为四类：

1. 验证语义切片算法在正常和异常场景下的行为稳定性。
2. 验证结构分片与语义分片组合后的 pipeline 行为正确。
3. 验证最终 chunk 向量化阶段的批处理和缓存行为。
4. 验证从 `markdown_parser` 到 `splitter` 的端到端链路可运行，并覆盖主要 Markdown 类型。

## 3. 当前测试文件

当前测试主要位于：

- `tests/core/splitter/test_semantic_chunker.py`
- `tests/core/splitter/test_pipeline_chunker.py`
- `tests/core/splitter/test_embedding_pipeline.py`
- `tests/core/splitter/test_markdown_parser_to_splitter_integration.py`

配套测试资源位于：

- `tests/core/splitter/fixtures/full_markdown_pipeline_fixture.md`
- `tests/core/splitter/artifacts/markdown_parser_to_splitter_visualization.md`

## 4. 测试分层

### 4.1 单元测试

单元测试关注单一策略或组件本身的输入输出行为。

当前单元测试包括：

- `PercentileSemanticChunker`
- `StructuredSemanticChunker`
- `ChunkEmbeddingPipeline`

### 4.2 轻量集成测试

轻量集成测试关注多个组件拼接后的行为是否正确，但通过 mock 屏蔽真实外部依赖。

当前主要是：

- `markdown_parser -> splitter -> embedding pipeline`

### 4.3 端到端链路验证

端到端测试不连接真实外部服务，而是尽量使用真实本地逻辑、Mock 外部网络调用，验证完整处理链是否走通。

当前覆盖链路：

```text
Markdown 文件
-> MarkdownParser.parse_file()
-> ImageDescriber / TableDescriber (mock)
-> StructuredSemanticChunker
-> ChunkEmbeddingPipeline
-> 可视化 artifact
```

## 5. Mock 策略

当前测试遵循一条原则：

- 本地逻辑尽量真实执行
- 网络或外部模型能力全部用 mock 替代

### 5.1 图片理解 Mock

图片描述使用 `MockVisionClient` 替代真实 VLM。

目的：

- 不依赖外部多模态接口
- 保证测试可重复
- 验证图片描述回填逻辑

### 5.2 表格总结 Mock

表格总结使用 `MockTableClient` 替代真实 LLM。

目的：

- 验证表格增强管线是否工作
- 避免外部模型调用导致的不稳定

### 5.3 Embedding Mock

语义切片与最终向量化使用多种 mock embedder：

- 基础 deterministic embedder
- 可控断点行为的 hybrid mock embedder
- 故意抛错的 failure mock embedder

目的：

- 精确控制语义距离分布
- 验证 fallback 分支
- 验证最终 embedding 调用次数与批次行为

### 5.4 Tokenizer Mock

测试中的 tokenizer 主要采用按词计数的 mock 实现。

目的：

- 避免依赖真实 tokenizer
- 让 token 规模和断点判断可预测
- 便于构造超长文本测试

## 6. 当前覆盖范围

### 6.1 `test_semantic_chunker.py`

重点覆盖：

- 动态分位阈值能触发语义断点
- 当前块低于最小 token 限制时，断点会被忽略
- 达到最大 token 上限时会强制切分
- overlap 能在强切场景中保留
- embedding 调用失败时会回退到长度切分

这组测试主要保证：

- 百分位语义切片算法本身稳定
- 关键工程保护机制有效

### 6.2 `test_pipeline_chunker.py`

重点覆盖：

- 先规则分片，再语义细分的两阶段流程
- 普通正文块在超长时进入语义切片
- 结构性块仍按既定策略输出

这组测试主要保证：

- `StructuredSemanticChunker` 的组合逻辑正确

### 6.3 `test_embedding_pipeline.py`

重点覆盖：

- 最终 chunk 会被再次向量化
- 相同内容重复处理时缓存可复用
- 向量化结果与 chunk 顺序保持一致

这组测试主要保证：

- 最终 embedding 阶段的正确性和性能保护机制

### 6.4 `test_markdown_parser_to_splitter_integration.py`

重点覆盖：

- 真实 `MarkdownParser.parse_file()` 调用
- 所有关键 Markdown 类型的解析与分片链路
- 图片描述和表格总结的 mock 外部调用
- 超长正文触发语义切片
- isolated chunk 的上下文 overlap
- 最终生成 chunk 可视化文档

这组测试主要保证：

- 从解析到切片再到向量化的整条链路可工作
- 模块集成后不会因为局部修改导致主流程失效

## 7. Markdown Fixture 覆盖内容

当前端到端测试 fixture 已覆盖以下 Markdown 类型：

- Front Matter
- Heading
- Paragraph
- Blockquote
- List
- Code Block
- Table
- Image
- Inline image 所在段落
- Math Block
- Horizontal Rule

此外还包含一个专门用于语义分片压力测试的超长章节：

- `Semantic Pressure Test`

该章节被设计为：

- token 数显著超过 `max_chunk_tokens`
- 能稳定触发语义切片 embedding 调用

## 8. Overlap 相关验证

当前测试已明确验证两类 overlap 行为：

### 8.1 语义切片内部 overlap

在长度强切场景下，测试会检查后续 chunk 是否带有前序尾部上下文。

### 8.2 最终 chunk 邻接上下文 overlap

在两阶段 pipeline 中，测试会检查：

- isolated chunk 不再是纯孤立块
- 图片块等也会带邻接上下文
- 对应 metadata 中会记录实际应用的上下文 token 数

## 9. 可视化产物

当前端到端测试会生成一份可视化文档：

- `tests/core/splitter/artifacts/markdown_parser_to_splitter_visualization.md`

该文档主要用于人工复检，内容包括：

- fixture 元素覆盖摘要
- mock 调用摘要
- 最终 chunk 列表
- 每个 chunk 的策略、标题路径、上下文 token 信息
- 每个 chunk 的完整文本内容

这个 artifact 的价值在于：

- 出现切片异常时，便于快速人工定位
- 便于复核语义分片和 overlap 是否符合预期

## 10. 测试执行方式

本项目测试口径统一为 `pytest`。

### 10.1 安装命令

如果本地环境尚未安装 `pytest`，推荐使用以下命令之一：

```bash
pip install pytest pytest-asyncio
```

如果希望按项目开发依赖一次性安装，则使用：

```bash
pip install -e ".[dev]"
```

项目 `pyproject.toml` 已声明：

- `pytest`
- `pytest-asyncio`

### 10.2 推荐执行命令

按文件执行：

```bash
pytest tests/core/splitter/test_semantic_chunker.py -v
pytest tests/core/splitter/test_pipeline_chunker.py -v
pytest tests/core/splitter/test_embedding_pipeline.py -v
pytest tests/core/splitter/test_markdown_parser_to_splitter_integration.py -v
```

一次性执行 splitter 相关测试：

```bash
pytest tests/core/splitter -v
```

按关键字筛选执行：

```bash
pytest tests/core/splitter -k semantic -v
```

### 10.3 异步测试说明

项目已在 `pyproject.toml` 中配置：

- `asyncio_mode = "auto"`

这意味着：

- 异步测试函数可以直接使用 `async def`
- 由 `pytest-asyncio` 自动接管事件循环
- 不再需要 `unittest.IsolatedAsyncioTestCase`

## 11. 测试设计原则

### 11.1 优先覆盖关键行为，不追求表面行覆盖率

当前更关心：

- 结构边界是否被保护
- 语义断点是否合理
- fallback 是否可靠
- 元数据是否完整

### 11.2 避免真实外部依赖

测试不会依赖：

- 真实 embedding 接口
- 真实 VLM 接口
- 真实表格总结模型

这样可以确保：

- 本地可重复执行
- CI 稳定
- 不受网络状态影响

### 11.3 保持“单元 + 集成 + 人工复检 artifact”三层组合

这是当前测试体系的关键特征：

- 单元测试负责算法正确性
- 集成测试负责模块协作正确性
- artifact 负责人工可解释性

## 12. 测试空白

当前仍有一些覆盖较弱的点：

- `markdown_parser` 的独立单元测试尚未系统化拆分
- 更多异常 Markdown 变体尚未覆盖
- 多文档批处理场景未单独测试
- 更复杂的 embedding 返回异常形态覆盖有限
- 真实 tokenizer 接入前的兼容性还未验证

## 13. 结论

当前测试体系已经能够较稳定地保障以下核心能力：

- Markdown 结构解析
- 结构分片
- 百分位语义切片
- 最终 chunk 向量化
- 图片和表格增强链路
- overlap 与 metadata 的基本正确性

对于本模块来说，这套测试已经能够承担回归保护职责，并为持续迭代提供稳定基线。
